# FinSignal Lakehouse — Data Model

## 1. Purpose

This document defines the data model for the FinSignal Lakehouse MVP.

The MVP focuses on trade-position reconciliation and bounded event-window analytics. The core data entities are securities, prices, starting positions, trades, reported positions, filing-style events, and an injected issues manifest.

The data model supports three layers:

1. **Raw Layer** — JSONL payloads generated by the synthetic ledger generator.
2. **Bronze Layer** — minimally transformed Parquet/Delta-style tables with ingestion metadata.
3. **Silver Layer** — cleaned and validated entities.
4. **Gold Layer** — investigation-ready reconciliation and event analytics outputs.

## 2. Layering Strategy

### Raw Layer

Raw data is stored as JSONL files.

Raw data simulates API/event-style payloads and should remain close to the original generated format.

### Bronze Layer

Bronze preserves raw fields and adds ingestion metadata.

Bronze should not perform heavy business logic.

Bronze adds:

* `_ingested_at`
* `_source_file`
* `_bronze_load_id`
* `_raw_record_hash`

### Silver Layer

Silver enforces schemas, normalizes fields, validates records, and prepares data for reconciliation.

Examples:

* normalize dates and timestamps
* standardize trade side values
* validate security IDs
* detect duplicate trade IDs
* detect missing prices
* validate quantities and prices

### Gold Layer

Gold contains investigation-ready outputs.

Examples:

* reconstructed positions
* reconciliation breaks
* trade quality flags
* event-window metrics

# 3. Raw and Bronze Entities

## 3.1 securities

### Grain

One row per tradable security.

### Raw Path

`data/raw/securities/securities.jsonl`

### Bronze Table

`bronze_securities`

### Columns

| Column       | Type    | Description                  |
| ------------ | ------- | ---------------------------- |
| security_id  | string  | Internal security identifier |
| ticker       | string  | Ticker symbol                |
| company_name | string  | Company name                 |
| exchange     | string  | Exchange name                |
| sector       | string  | Business sector              |
| is_active    | boolean | Whether security is active   |

### Bronze Metadata

| Column           | Type      | Description                          |
| ---------------- | --------- | ------------------------------------ |
| _ingested_at     | timestamp | Time record was ingested into bronze |
| _source_file     | string    | Source file path                     |
| _bronze_load_id  | string    | Unique load identifier               |
| _raw_record_hash | string    | Hash of raw record values            |

## 3.3 starting_positions

### Grain

One row per account, security, and baseline position date.

### Raw Path

`data/raw/starting_positions/starting_positions.jsonl`

### Bronze Table

`bronze_starting_positions`

### Columns

| Column        | Type   | Description                  |
| ------------- | ------ | ---------------------------- |
| account_id    | string | Account identifier           |
| security_id   | string | Internal security identifier |
| position_date | date   | Baseline position date       |
| quantity      | long   | Starting share quantity      |

### Notes

Starting positions are trusted checkpoint baselines.

The MVP does not reconstruct full account history from inception. It reconstructs expected positions over the reconciliation window using:

`starting quantity + cumulative buys - cumulative sells`

Negative position quantities are allowed in the synthetic ledger and represent short positions. The reconciliation engine validates position consistency, not long-only portfolio constraints.

# 4. Silver Target Tables

## 4.1 silver_securities_clean

Cleaned security reference table.

Expected validation:

* `security_id` is not null
* `ticker` is not null
* duplicate `security_id` records are flagged

## 4.2 silver_prices_clean

Cleaned price table.

Expected validation:

* `security_id` exists in securities
* `price_date` is valid
* prices are positive
* `high_price >= low_price`
* duplicate security/date rows are flagged

## 4.3 silver_starting_positions_clean

Cleaned starting position baselines.

Expected validation:

* every traded account/security pair has a starting position
* missing starting positions should be explicitly represented as quantity 0
* quantities are numeric

## 4.4 silver_trades_clean

Cleaned trade table.

Expected validation:

* `trade_id` is not null
* `account_id` is not null
* `security_id` exists in securities
* `side` is BUY or SELL
* `quantity` is positive
* `execution_price` is positive
* duplicate trade IDs are flagged
* late-arriving trades are flagged
* missing price records are flagged

## 4.5 silver_reported_positions_clean

Cleaned reported positions.

Expected validation:

* `account_id` is not null
* `security_id` exists in securities
* `position_date` is valid
* `reported_quantity` is numeric

## 4.6 silver_events_clean

Cleaned filing-style event table.

Expected validation:

* `event_id` is not null
* `security_id` exists in securities
* `event_date` is valid

| Column             | Description                                                                                            |

## 5.4 gold_event_window_metrics

### Grain

One row per event and security.

### Example Fields

| Column                       | Description                  |
| ---------------------------- | ---------------------------- |
| event_id                     | Event identifier             |
| security_id                  | Security identifier          |
| event_date                   | Event date                   |
| event_type                   | Filing-style event type      |
| pre_event_return             | Return before event          |
| post_event_return            | Return after event           |
| pre_event_volatility         | Volatility before event      |
| post_event_volatility        | Volatility after event       |
| position_change_around_event | Position change around event |
| exposure_change_around_event | Exposure change around event |

### Notes

Event-window joins must be bounded using trading-day indexes or Spark window functions.

The project should avoid unbounded event-to-price joins.
